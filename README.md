# TP1_EDA
Este programa está pensado para permitir a una compañía de turismo la administración práctica de las reservas. Usa una lista doblemente enlazada donde cada reserva mantenga el nombre de la persona, un ID único y el precio. Esta es la estructura con la que el usuario puede realizar operaciones básicas: añadir una reserva, buscarlas por su ID, eliminarlas, actualizar los datos y ver todas. Por si eso fuera poco, el sistema también proporciona estadísticas tales como el promedio de los costos, valor mínimo, máximo, media y moda. Además permite ordenar la lista según el nombre del cliente o según el precio; de forma creciente o decreciente. Por qué elegir una lista doblemente enlazada? Porque necesitamos insertar y eliminar elementos sin conocer el tamaño total del conjunto o con uno que pueda cambiar. Evitamos también desplazamientos en arreglos. Adicionalmente, debido a que es posible atravesarla en ambos sentidos, se puede calcular la mediana y hacer el reajuste de punteros más fácilmente. A pesar de que en la implementación presente las inserciones son realizadas siempre al final, la estructura estará abierta a cambios futuros para poder insertar en cualquier lugar. El diseño del código está así por dos razones principales: En cuanto a la visibilidad de atributos, se han hecho con alcance de paquetes, porque hace mas simple su acceso desde la lista... y porque es la academia: ni getters y setters. Sin embargo, se agregaron algunas validaciones importantes: Si alguien intenta ingresar un costo negativo, automáticamente es corregido a cero y se notifica al usuario. Además, antes de introducir una nueva reservación, se realiza la verificación de que ya existe alguna otra con el mismo id para garantizar su unicidad.
Para organizar los datos, se utilizó el algoritmo merge sort, que es bastante eficiente para listas enlazadas (O(n log n)) y, además, es estable. Para encontrar el punto medio de la lista sin saber su tamaño, se aplica la técnica de punteros lento y rápido. En el proceso de ordenación por costo, si dos reservas tienen el mismo valor, se desempata utilizando el nombre del cliente, lo que permite un orden más predecible. Después de cada ordenación, se actualiza la referencia al último nodo (tail), ya que la cabeza puede cambiar y es importante que la cola esté correctamente apuntada. Las estadísticas se obtienen al recorrer la lista una vez, lo que permite calcular la suma, el mínimo, el máximo y la cantidad de elementos. La mediana se calcula asumiendo que la lista ya está ordenada por costo, y se muestra una advertencia si no es así. Para obtener la moda, se utiliza un doble bucle que cuenta las frecuencias; aunque no es el método más óptimo para conjuntos muy grandes, es suficiente para el volumen esperado de reservas. El menú de interacción es bastante sencillo, basado en un bucle que se repite hasta que el usuario decide salir. Incluye validación de entrada para evitar que el programa se cierre abruptamente si se ingresan caracteres no numéricos. En cuanto a la complejidad, la mayoría de las operaciones (buscar, insertar con verificación, eliminar, actualizar, mostrar) son lineales O(n) debido a los recorridos necesarios. El ordenamiento alcanza O(n log n), mientras que el cálculo de la moda es O(n²), lo cual es aceptable para un número moderado de reservas. Durante el desarrollo, se implementaron varias mejoras: se añadió la validación de costo negativo también en la actualización, se ajustó el cálculo de la mediana para listas con un número par de elementos, se unificaron las comparaciones de texto usando compareToIgnoreCase para evitar diferencias por mayúsculas, y se creó un método que actualiza la cola después de ordenar. También se incorporaron controles para listas vacías y manejo de entradas inválidas, lo que hace que el programa sea más robusto y fácil de usar.Durante el proceso de diseño del programa se realizaron numerosas actualizaciones y revisiones para mejorar la seguridad y la usabilidad. Por ejemplo, se modificó la función de actualización del programa para incluir la validación de costes negativos; anteriormente, un fallo en la validación podía dar lugar a errores en la precisión de los datos. Además, también se modificó el cálculo del promedio para los casos en que hay un número par de elementos en una lista; este cambio resultó en una mayor precisión del cálculo del promedio. Para evitar que el texto se escriba en mayúsculas o minúsculas (debido a la variación en el uso de mayúsculas y minúsculas), la función de comparación de texto normaliza el texto mediante el método compareToIgnoreCase. Tras ordenar la cola, se añadió un método que permite actualizarla y mejorar su organización en el sistema. También se añadieron validaciones para gestionar correctamente las listas vacías y se mejoró el método de gestión de entradas no válidas, lo que aumenta la robustez y durabilidad del programa.
